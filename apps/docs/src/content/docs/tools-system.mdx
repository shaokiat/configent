Tools are the extension points that let the agent do more than retrieve text. Each client YAML lists the tools available to its agent; the agent loop calls them as needed during a turn.

---

## Architecture

```
config/acme-fab.yaml            tools/registry.py
  agent:                          _REGISTRY = {
    tools:                          "search_docs":    (DEFINITION, execute),
      - search_docs         →       "get_document":   (DEFINITION, execute),
      - get_document                "pricing_lookup": (DEFINITION, execute),
      - pricing_lookup              "coverage_check": (DEFINITION, execute),
                                  }
                                       │
                                  validated at startup
                                  (unknown names → ValueError, not 404)
```

The registry is a plain dict — no dynamic loading, no plugin system. Adding a tool means writing a module and adding one entry to `_REGISTRY`.

## Built-in tools

### `search_docs` (shared)

Semantic similarity search over the client's corpus. The primary retrieval tool — the agent is instructed to call it before answering any factual question.

```python
# Input
{"query": "PX900 chamber seal replacement interval", "k": 5}

# Output — Anthropic search_result blocks (enables inline citations)
[
  {
    "type": "search_result",
    "source": "corpus://acme-fab/spare-parts-catalog",
    "title": "Spare Parts Catalog",
    "content": [{"type": "text", "text": "...chunk text..."}],
    "citations": {"enabled": True}
  },
  ...
]
```

Empty retrieval returns a plain `text` block telling the model no documents matched — the model is instructed to say it doesn't know rather than hallucinate.

### `get_document` (shared)

Fetches the full text of a document by `document_id`. Used when a search chunk lacks enough context — for example, when a procedure spans multiple sections.

```python
# Input
{"document_id": "a3f2b1c4"}

# Output
{"document_id": "a3f2b1c4", "title": "...", "source_uri": "...", "text": "..."}
```

Enforces `client_id` isolation — a document belonging to a different client returns a not-found error even if the ID is valid.

### `pricing_lookup` (acme-fab)

Looks up unit price, volume discounts, and lead time for an Acme part number. Backed by a hardcoded catalog in the current implementation.

### `coverage_check` (meridian)

Checks whether a specific procedure or treatment is covered under a Meridian Insurance policy.

## Tool lifecycle in the agent loop

When the model decides to call a tool, it returns a `tool_use` block in its response:

```json
{
  "type": "tool_use",
  "id": "toolu_01A2B3",
  "name": "search_docs",
  "input": {"query": "PX900 seal replacement interval", "k": 5}
}
```

The loop executes the tool and wraps the result in a `tool_result` block, which is appended as a `user` message for the next API call:

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01A2B3",
  "content": [{"type": "search_result", ...}]
}
```

The `tool_use_id` must match the `id` from the model's block — the API validates this. Tools within a single iteration run concurrently via `asyncio.gather`. A tool that raises returns `{"error": "<message>"}` rather than crashing the turn.

## Adding a new tool

**1. Create the module** under `tools/<client_or_shared>/`:

```python
DEFINITION = {
    "name": "my_tool",
    "description": "...",
    "input_schema": {
        "type": "object",
        "properties": {
            "param": {"type": "string", "description": "..."},
        },
        "required": ["param"],
    },
}

async def execute(tool_input: dict, *, client_id: str, db) -> dict | list:
    return {"result": do_something(tool_input["param"])}
```

**2. Register it** in `tools/registry.py`:

```python
_REGISTRY = {
    ...
    "my_tool": (my_tool.DEFINITION, my_tool.execute),
}
```

**3. Add it to a client config:**

```yaml
agent:
  tools:
    - search_docs
    - my_tool
```

Unknown tool names in YAML fail at startup with a clear error — validation happens at config load time, not at request time.

## Return value conventions

| Return type | How it's sent to the model |
|-------------|---------------------------|
| `dict` or `list[non-block]` | `json.dumps(result)` as a text string in `tool_result.content` |
| `list[{"type": ..., ...}]` | Passed verbatim as `tool_result.content` blocks (enables `search_result` citation blocks) |

Use the content-block form when you need the model to emit citations. Use plain JSON for everything else.

## Tool definitions and caching

Tool definitions are sent with every API call, sorted alphabetically before inclusion. This ensures the tool list serializes identically regardless of YAML order — required for Anthropic's prefix-match prompt cache to hit. See [Prompt Caching](/configent/docs/prompt-caching) for details.
