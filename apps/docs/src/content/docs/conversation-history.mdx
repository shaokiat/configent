Every turn loads the full prior message history from Postgres, appends it to the API request, and persists new messages after the loop completes.

---

## Data model

```
conversations
  id          UUID PK
  client_id   VARCHAR(64)   — scopes the conversation to one client
  started_at  DATETIME
  total_cost  FLOAT         — accumulated USD across all turns
  total_tokens INT

messages
  id              INTEGER PK autoincrement
  conversation_id UUID FK → conversations.id (CASCADE DELETE)
  role            VARCHAR(32)   — "user" | "assistant"
  content         JSON          — raw Anthropic content block list
  citations       JSON          — { segments: [...] } on final assistant messages
  created_at      DATETIME
```

Messages are stored in raw Anthropic wire format — `tool_use` and `tool_result` blocks are preserved verbatim, no lossy serialization.

---

## Creating a conversation

On the first turn, pass `conversation_id: null`. The API creates a new `Conversation` row and returns the UUID in the `done` SSE event:

```json
// POST /api/c/acme-fab/chat/stream
{ "message": "What is the seal replacement interval?", "conversation_id": null }

// done event
{ "conversation_id": "3f2a1b4c-...", "input_tokens": 1240, ... }
```

Include the returned `conversation_id` on every subsequent request to continue the conversation.

---

## Loading history

At the start of each turn, `_prepare_conversation()` loads all prior messages ordered by `id` and prepends them to the request:

```python
history = SELECT messages WHERE conversation_id = ? ORDER BY id
messages = history + [{"role": "user", "content": new_user_message}]
```

The full history is sent on every turn — no sliding window or summarisation. As conversations grow, prompt caching absorbs the cost of resending historical messages. See [Prompt Caching](/configent/docs/prompt-caching).

---

## Persisting a turn

After the agent loop completes, `_persist_turn()` writes everything added during the turn — everything after `history + user message`:

- Intermediate assistant messages with `tool_use` blocks
- Intermediate user messages with `tool_result` blocks
- The final assistant message (gets a `citations` field with the frontend-friendly `segments` list)

On any exception, the session rolls back — the conversation history retains all messages from prior turns.

---

## Multi-client isolation

`conversation_id` is a global UUID, but every history query scopes to `client_id` via the `Conversation` row's foreign key. A conversation started under `acme-fab` cannot be continued as `meridian-insurance` — the router validates `client_id` against the config registry on every request.

---

## Request flow

```
POST /api/c/:client_id/chat/stream
        │
        ├─ _prepare_conversation(db, client_id, conversation_id)
        │       if None   → INSERT conversation, return []
        │       if set    → SELECT messages ORDER BY id, return history
        │
        ├─ INSERT user message
        │
        ├─ agent loop (1–8 iterations)
        │       each call: [system] + [tools] + history + [new user msg] + [tool turns]
        │       tool turns accumulate in-memory; not persisted until commit
        │
        ├─ _persist_turn(db, conversation_id, history_len, result)
        │       INSERT assistant + tool_result messages
        │       final assistant message gets citations attached
        │
        └─ db.commit() → yield done event
```

---

## Querying history directly

```sql
SELECT role, created_at, jsonb_pretty(content::jsonb)
FROM messages
WHERE conversation_id = '3f2a1b4c-...'
ORDER BY id;
```
