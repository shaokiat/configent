Documents in `corpora/<client_id>/` are parsed, chunked, embedded, and stored in pgvector. At query time, the user's question is embedded and matched against those chunks via cosine similarity — this is the full RAG loop.

```
corpora/<client_id>/
  ├── manual.pdf
  ├── runbook.md
  └── spec.html
        │
        ▼  parse_document()       → ParsedDocument { title, text }
        ▼  chunk_document()       → list[Chunk]     (token-aware, overlapping)
        ▼  embed(texts)           → list[list[float]] (Voyage AI voyage-3, 1024-dim)
        ▼  db.add(Document, Chunk rows)
        ▼  db.commit()
```

All four stages run inside a single `AsyncSession` and commit atomically per document. A failed embed on one document rolls back that document only — already-committed documents are unaffected.

---

## Stage 1 — Parse

`retrieval/parsers.py` maps file extensions to parsers:

| Extension | Parser | Title extraction |
|-----------|--------|-----------------|
| `.pdf` | pypdf page extraction | `path.stem` (titlecased) |
| `.md`, `.markdown` | plain text read | First `# heading`, else `path.stem` |
| `.html`, `.htm` | BeautifulSoup, strips `<script>`/`<style>` | `<title>` → `<h1>` → `path.stem` |

Unsupported extensions are silently skipped by `iter_corpus()`. All parsers return a `ParsedDocument(title, text, source_path)`.

---

## Stage 2 — Chunk

`retrieval/chunker.py` splits text into token-aware overlapping chunks using `tiktoken` (cl100k_base):

**Split strategy (in priority order):**
1. Double-newline (paragraph boundaries)
2. Sentence boundaries, if a paragraph exceeds `chunk_size`
3. Hard token split, if a single sentence still exceeds `chunk_size`

The last `overlap` tokens from the previous chunk are prepended to the next, ensuring sentences that straddle a boundary appear in full in at least one chunk.

```yaml
corpus:
  chunking:
    chunk_size: 800   # tokens per chunk (default: 800)
    overlap: 100      # overlap tokens between chunks (default: 100)
```

---

## Stage 3 — Embed

`retrieval/embed.py` calls Voyage AI's `voyage-3` model (1024 dimensions):

- Documents use `input_type="document"`, queries use `input_type="query"` — asymmetric encoding improves retrieval quality.
- Batches of up to 128 texts per API call.
- 429 and 5xx errors retry with exponential backoff, up to 6 attempts starting at 21s (respects Voyage's per-minute window).

---

## Stage 4 — Upsert

Each document is keyed by `MD5(client_id + source_uri)[:8]` and fingerprinted with `SHA256(text)[:32]`:

| State | Action |
|-------|--------|
| New document | Insert |
| Existing, same content hash | Skip |
| Existing, changed content hash | Delete + re-insert (cascade deletes chunks) |

Only changed files are re-processed and re-embedded on subsequent runs.

## Search

`retrieval/search.py` handles the retrieval side. When `search_docs` is called, it:

1. Embeds the query with `input_type="query"` (asymmetric from `input_type="document"` — improves retrieval quality)
2. Runs a pgvector cosine similarity query scoped to `client_id`, fetching `k×2` candidates
3. Drops hits below a similarity floor of `0.3` — irrelevant results never reach the model
4. Returns the top `k` hits

```python
async def search(db, client_id, query, k=5, floor=0.3) -> list[Hit]:
    query_vec = await embed_query(query)
    # cosine similarity = 1 - cosine distance (<=>)
    rows = SELECT chunks JOIN documents
           WHERE client_id = ?
           ORDER BY similarity DESC
           LIMIT k * 2

    return [hit for hit in rows if hit.similarity >= floor][:k]
```

Each `Hit` carries `document_title`, `source_uri`, `text`, `similarity`, and `chunk_index` — enough for `search_docs` to build the `search_result` content blocks the citation API needs.

## Database schema

```
documents
  id            VARCHAR(64)   PK  — MD5(client_id:source_uri)[:8]
  client_id     VARCHAR(64)       — foreign key → clients.id
  source_uri    VARCHAR(512)      — corpus://<client_id>/<stem>
  title         VARCHAR(512)
  content_hash  VARCHAR(64)       — SHA256(text)[:32]; drives skip/replace logic
  ingested_at   DATETIME

chunks
  id            INTEGER       PK  autoincrement
  document_id   VARCHAR(64)       — FK → documents.id (CASCADE DELETE)
  client_id     VARCHAR(64)       — denormalized for index-scoped search
  text          TEXT
  embedding     VECTOR(1024)      — pgvector column
  chunk_index   INTEGER
  metadata      JSON              — { document_title, source_uri, position }
```

`client_id` on `chunks` is denormalized intentionally — pgvector similarity queries filter on it directly, avoiding a join to `documents` on the hot path.
